# Sanitized Corrected-Copy Chain and Ledger Manifest

No production identifiers, file names, borrower data, storage keys, or document contents are
included in this manifest.

## Accepted chain

1. Current generated legal document remains the independent renderer-owned original.
2. Initial staff signed copy retains one file, uploader, request/action identity, upload audit,
   legal audit, workflow, and version body; it has no predecessor and resolves no correction.
3. A correction/return retains one exact item/document or approval-stage target and one matching
   actor, authorising role, audit, workflow, and version body.
4. The corrected copy must directly extend the current signed-copy tail, target that review action,
   and retain the same complete file/owner ledger contract.
5. Sequential corrections extend the same immutable chain; every historical row remains present,
   while the sole current tail and every consumed resolution remain coherent.

## Fail-closed probes

- File checksum, uploader, request/action id, remarks, resolution target, upload audit, legal audit,
  workflow, version, and current renderer changes restore the blocker.
- Missing predecessor, duplicate audit/upload/version evidence, null target, invalid condition
  evidence, and ambiguous open reviews cannot clear correction truth.
- Primary/governed multi-role actors retain the exact checklist or approval-stage authorising role;
  a later-stage condition/return is not attached to Company Secretary history.

The executable assertions and terminal output are retained in `terminal-logs/`.
