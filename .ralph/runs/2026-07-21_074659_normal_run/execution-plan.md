# Execution Plan

Selected slice: 010K2-loan-ledger-statements-and-export

## Delivery boundary

Add backend-only CSV loan-ledger statement request, status, and signed-download actions. The statement
module will read rows exclusively through the canonical `processes.loan_servicing` selector, use the
existing scheduler row as export-job truth, retain generated bytes through `DocumentFile` storage, and
record metadata-only audit events. No frontend, duplicate ledger, or duplicate financial calculation is
in scope.

## Deep-module seam

- Public interface: request a statement for one scoped loan/date range, read that job's status, and
  consume a short-lived signed download capability.
- Hidden implementation: canonical selector pagination, opening/closing reconciliation, deterministic
  CSV rendering, checksum verification, scheduler lifecycle, document retention, masking, capability
  binding, and audit metadata.
- Existing adapters: `ScheduledJob`, `LocalDocumentStorage`, canonical loan read/portal scope, and the
  central `AuditLog` writer conventions.

## TDD tracer bullets

1. RED/GREEN: staff request produces a completed, parseable CSV whose partial-range rows and
   opening/movement/closing totals reconcile with the canonical ledger selector.
2. RED/GREEN: empty/full-history ranges, stable ordering, generated-at/as-of metadata, reversals, and
   idempotent replay preserve exactly the same job/artifact/checksum.
3. RED/GREEN: status and download require loan read plus export authority and exact object/job scope;
   borrower access is restricted to the authenticated portal member's own loan and masked projection.
4. RED/GREEN: signed capabilities are short-lived and bound to actor, job, loan, and checksum; expired,
   superseded/revoked, cross-user, cross-loan, and guessed-job attempts fail without disclosure.
5. RED/GREEN: request, generation, download success, and denial audits contain references/checksums but
   never rows, raw bank data, storage keys, or signed URLs.
