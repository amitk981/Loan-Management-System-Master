## Backend/API Scope
1. Add `GET /api/v1/document-files/{document_id}/download/` from `api-contracts.md` §26.2.
2. Prefer Response Option A for MVP/local tests: signed or time-limited download descriptor
   `{download_url, expires_at}` generated by the storage adapter. If local adapter cannot
   produce a true signed URL, return a deterministic local adapter URL/token shape and document
   it in `API_CONTRACTS.md`.
3. Reuse the `DocumentFile` model, `storage_provider = "local"`, and `storage_key` written by
   003C. Extend the 003C storage boundary if needed; do not create new document metadata tables.
4. Do not implement loan-document checklist approvals, template generation, streaming binary
   downloads, or frontend document UI in this slice.

## Database/Model Impact
No schema change expected. If a short-lived download-token table is introduced, justify it in
`execution-plan.md`, keep it non-destructive, and add expiry cleanup tests.

## API Contracts
Update `docs/working/API_CONTRACTS.md` with the download response option chosen, expiry
window, auth/permission errors, not-found behavior, and audit action.

## Permissions
Require session-bound bearer auth. Use the source-mapped `documents.file.download` permission
from `docs/source/auth-permissions.md`; do not broaden the 003C upload permission into download
permission. Keep sensitivity-specific object rules as an explicit ASSUMPTIONS entry until the
source documents define the role/sensitivity matrix in implementable detail.

## Audit Requirements
Successful download-link creation must write one `AuditLog` row for document download/access:
actor user, `entity_type = "document_file"`, `entity_id = document_id`, `action` naming the
download/access event, and new-value metadata containing filename/sensitivity/provider but no
signed secret or raw bytes. Failed auth/permission/not-found requests must not leak document
metadata.
Use a stable action name such as `documents.file.downloaded` unless the implementation records
only signed-link issuance, in which case document the chosen action in `API_CONTRACTS.md`.

## Validation Rules
- Unknown `document_id` returns standard `404 NOT_FOUND` without leaking storage details.
- Missing/malformed/revoked bearer token returns existing `401` envelopes.
- No-permission actor returns `403 PERMISSION_DENIED`.
- Restricted/confidential documents must follow the same permission path as internal/public
  until object-level source rules are added; record any sensitivity-specific gap in
  `ASSUMPTIONS.md`.

## Test Cases
- Backend TDD red/green: download endpoint fails before implementation.
- API: authorized user receives `{download_url, expires_at}` in the standard envelope.
- API: `401`, `403`, and `404` envelopes are standard and do not create audit rows.
- Audit regression: successful download-link creation writes exactly one audit row.
- Security regression: response never includes `storage_key`, checksum, raw bytes, or signed
  token secret beyond the intended URL field.
