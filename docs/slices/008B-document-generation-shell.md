## Goal
Generate and retain one immutable loan-document record from an approved 008A template and
authoritative application facts through the source §26.4 API, without implementing checklist,
signature, stamp, notarisation, custody, or verification workflows.

## User Value
Moves the platform one verifiable step closer to a working end-to-end lending system without broad module-sized changes.

## Backend/API Scope
1. Add `POST /api/v1/loan-applications/{loan_application_id}/loan-documents/generate/` with the
   standard envelope. Accept only `document_type`, `template_id`, and bounded `output_format`
   (`pdf` or `docx`/Word according to the source format vocabulary).
2. Resolve the exact 008A template by id. It must be approved, effective on generation date,
   match `document_type`, match the application's Individual/FPO borrower variant, and reference a
   valid template file. Template metadata or file id alone grants no download/generation authority.
3. Populate only merge fields declared by that retained template, from authoritative persisted
   application/member/nominee/witness/shareholding/appraisal/sanction facts. Missing required merge
   facts block generation with field errors; never substitute display fixtures, empty policy facts,
   or mutable live values for an immutable reviewed/sanctioned snapshot.
4. Render through a documents-owned generator/storage seam and persist the generated bytes through
   the existing storage adapter. The response returns `loan_document_id`, `document_type`,
   `generation_status=generated`, `document_id`, and deterministic safe `file_name`.
5. Persist immutable application/template/generated-file linkage plus §16.3 document category,
   required party, initial execution/verification states, and creation time. Leave loan account,
   stamp, notarisation, custody, retention, and verifier fields unset for their owning slices.
6. Add `GET /api/v1/loan-applications/{loan_application_id}/loan-documents/` with standard bounded
   pagination and object-scoped metadata only. Do not add download, verify, signature, checklist,
   stamp, or notarisation actions.
7. The Term Sheet merge projection must be capable of the 13 SOP §4.9 facts from the digest.
   Loan Agreement generation is blocked until the exact application's Term Sheet is executed; this
   slice may expose the guard seam but must not invent execution evidence owned by later slices.

## Database/Model Impact
Add one non-destructive `loan_documents` migration for the §16.3 fields needed above, with indexes
on application, document type/category, generation/execution/verification status, and protected
nullable FKs to the retained template and generated document file. Do not add later workflow tables.

## API Contracts
Create or update the API contract for this capability.

## Permissions
Require `documents.loan_document.generate` plus existing loan-application object access for POST.
Require `documents.loan_document.read` plus the same application scope for GET. Template read,
document-file read/download, verify, or global settings permissions do not imply generation.

## Audit Requirements
Successful generation writes attributable audit/workflow evidence containing application,
template/version, document type/format, generated document-file id, and request trace—not rendered
borrower content. Denials and validation failures create no loan document, file, or success event.

## Validation Rules
- Application must be sanction-approved and object-readable by the actor; other states return the
  canonical invalid-state contract without reading unrelated applicant/template facts.
- Template approval/effective/type/borrower-variant/file checks are mandatory and fail closed.
- Merge fields are template-declared unique names and every required field must resolve from the
  authoritative application aggregate; unknown merge fields are validation errors.
- Generated file names are safe and deterministic from document type plus application reference;
  no borrower name or sensitive identifier appears in the file name or audit row.
- A repeated exact request must not create ambiguous duplicate evidence: return the retained result
  or a stable conflict according to the repository's existing replay convention, recorded explicitly
  in ASSUMPTIONS.md if the cited source remains silent.

## Test Cases
- Generate/list a PDF and Word document using the exact approved/effective Individual and FPO
  variant; assert retained template id/version, generated-file linkage, initial states, and response.
- Term Sheet tracer proves all 13 source facts come from persisted authoritative data; missing or
  unknown merge facts produce zero writes.
- Reject draft/retired/not-yet-effective/expired/wrong-type/wrong-borrower/missing-file templates,
  unsanctioned applications, unsafe formats, inaccessible objects, and missing permission.
- Reader/generator/template-manager/document-download permission matrix proves no authority leaks;
  list totals/pagination are object-scoped before serialization.
- Replay/concurrent generation produces one canonical retained result and one attributable evidence
  ledger; generation failure leaves neither metadata nor orphaned stored bytes.
- Architecture regression: generation uses the document-owned seam and does not import private
  models from member/credit/approval modules for business decisions.
