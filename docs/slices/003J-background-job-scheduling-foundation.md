## Goal
Add a minimal backend background-job scheduling foundation that later slices can use for reminders,
async reports, and notification generation without sending real communications or running external
workers in this slice.

## User Value
Moves the platform one verifiable step closer to a working end-to-end lending system without broad module-sized changes.

## Backend/API Scope
Implement only a local scheduler/job metadata shell:
- A model for scheduled jobs with source-neutral fields: job id, job type/code, status, due/run time,
  related entity type/id when applicable, attempts, last error summary, created/updated timestamps.
- A small service boundary to enqueue a job idempotently and mark it running/succeeded/failed for
  future modules.
- No Celery/Redis/cloud scheduler dependency unless already pinned and importable.
- No real email/SMS/courier delivery, no reminder generation business rules, no report generation,
  and no automatic notification creation beyond what 003IA already does for communication sends.

## Database/Model Impact
Non-destructive model/migration changes for this capability, if needed.

## API Contracts
If an endpoint is added, record it in `docs/working/API_CONTRACTS.md` with standard envelopes,
strict query validation, and top-level pagination. Keep it clearly separate from
`/api/v1/dashboard/`, `/api/v1/notifications/`, and `/api/v1/communications/`.

## Permissions
Use the narrowest existing administrative/ops permission. If source docs do not name a scheduler
permission, record the assumption in `ASSUMPTIONS.md`; do not reuse communication-send or dashboard
permissions for scheduler administration.

## Audit Requirements
Audit metadata-only scheduler mutations if exposed through an API. Do not audit raw rendered
message bodies, provider payloads, or report contents.

## Validation Rules
Validate job type/status values, related entity UUIDs, idempotency keys if supported, and legal
state transitions. Do not invent reminder timing, report cadence, or notification business rules.

## Test Cases
- Service tests for enqueue idempotency and legal/illegal status transitions.
- API tests, if an endpoint is added, for `401`, `403`, strict query validation, pagination, and no
  provider side effects.
- Regression test that dashboard tasks and notification inbox rows are not generated by the
  scheduler shell in this slice.
