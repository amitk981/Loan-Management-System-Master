# Prototype Gap Report

The prototype is useful as visual and workflow reference, not production truth.

| Gap | Impact | Where Seen | Recommended Handling |
|---|---|---|---|
| Most production screens still use mock data | Frontend/backend contract mismatch risk | `sfpcl-lms/src/data/mockData.ts` and module pages not listed as API-backed in `PROTOTYPE_INVENTORY.md` | Continue replacing one vertical slice at a time. Do not add mock fallbacks to screens already wired to backend APIs. |
| Backend implementation is partial | Cannot validate later module business rules server-side yet | Modules beyond completed auth/admin/audit/document/config/communication/dashboard foundations | Each backend slice must start from source docs and tests, then expose only source-backed behavior. |
| Demo role handling | Not production-grade auth/RBAC | `RoleContext`, login/dashboard | Implement auth and permission tests before real access control. |
| Module test coverage is uneven outside completed slices | Low confidence for future mock-to-API replacements | Prototype-only module pages | Add focused frontend/backend tests with each vertical slice before replacing mock behavior. |
| Missing error/loading/empty states | User experience and API readiness risk | Many screens | Include state coverage in frontend/API slices. |
| Financial and compliance rules in UI/mock form | High-risk drift from source docs | Appraisal, sanction, repayment, default | Move to backend/service layer with tests when backend exists. |

## Major Conflict Policy
If prototype behavior conflicts with `docs/source/`, source docs win. Record conflicts in `ASSUMPTIONS.md`, an ADR, or a slice blocker.
